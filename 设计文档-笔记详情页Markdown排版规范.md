## Problem Statement

大厂笔记产品的详情页，内容来自三个渠道：用户手写、AI 生成、外部导入。三种来源的 Markdown 在 WebView 里渲染风格不统一，用户体验割裂。设计师（東东）已经出了一版基于 HyperOS 设计系统的排版规范（420 版），需要：
1. 校对规范缺陷，补齐缺失元素
2. 用真实 Markdown 渲染引擎跑一个 Demo，验证规范端到端可行性
3. Demo 通过后推动研发全量实现

## Status Quo

研发已有 Markdown 渲染实现，但没有统一的排版规范。三种来源的不一致具体表现为：
- **用户手写内容**：格式简单（标题 + 正文 + 列表为主），但研发用默认样式渲染，行高、间距没有设计感
- **AI 生成内容**：格式丰富（常含表格、代码块、任务列表、嵌套引用），但这些元素缺少统一样式定义，渲染效果参差
- **导入内容**：来自其他编辑器/平台，可能携带非标准 Markdown 语法或内联 HTML，渲染结果不可预测

核心矛盾：三种来源的内容混排在同一个页面里，用户能明显感知到「这段和那段不是一个风格」。

## Target User & Narrowest Wedge

直接用户：研发团队（拿到规范后实现）
最终用户：笔记产品的所有使用者
最小验证单元：一个可以粘贴任意 Markdown 内容的实时预览页，覆盖 Light/Dark 模式切换

## Constraints

- 基于 HyperOS 设计系统
- 字体：MiSans（简中 v4.009）+ MiSans Latin（英文 v4.007），Demo 中使用系统 misans 字体模拟
- 8dp 间距体系
- 必须支持 Light/Dark 双模式
- WebView 渲染环境
- 渲染性能：需考虑长文档场景（数千行 Markdown），Demo 阶段不做性能优化但需标注潜在瓶颈
- 间距体系速查（原规范 8dp 体系）：xs=4, sm=8, md=12, lg=16, xl=24, xxl=32
- 行高规则：行高由 fontSize × multiplier 计算得出，结果不强制对齐 8dp 网格（行高是排版属性，非布局间距）
- Demo 字体策略：内嵌 MiSans woff2 子集（中文常用 3000 字 + 拉丁字母），确保 Demo 渲染效果与最终产品一致

## Premises

1. **PageTitle 不走 Markdown 解析** — 它是系统层注入的页面标题，和 Markdown 的 `#` 完全解耦。`#` 在正文中始终映射为 H1。本文档统一使用 PageTitle 指代原规范中的「H0」。
2. **MVP 阶段覆盖核心 Markdown 元素** — 包括表格、图片、链接、任务列表、代码块语法高亮。脚注和数学公式视 Open Questions 确认结果决定是否纳入 MVP。AI 生成和导入内容会包含这些元素，不覆盖就会回到「各自发挥」的老路。
3. **Demo 用真实 Markdown 解析引擎验证** — 静态截图无法验证多来源内容的一致性，必须能粘贴真实内容测试。
4. **成功标准是三合一** — 三种来源视觉统一 + 暗黑模式完美适配 + 研发零歧义交付。

## 规范缺陷清单（420 版校对结果）

### 必须修复

| # | 缺陷 | 影响 | 修复建议 |
|---|------|------|----------|
| 1 | PageTitle 和 H1 都映射到 `#`，消歧不清 | 研发无法区分 | 将 PageTitle 从 Markdown 映射中移除，明确标注「不走 Markdown 解析，由系统注入」。正文中 `#` 始终为 H1 |
| 2 | 缺少表格（table）样式定义 | AI 生成内容常含表格 | 补充：th 背景 code.bg，td padding 12dp，边框 1px block.divider 色，字号 text.primary 同款 16dp/1.6x |
| 3 | 缺少图片（image）渲染规则 | 导入内容常含图片 | 补充：max-width 100%，圆角 8dp，上下间距 md(12)，caption 用 text.secondary 13dp |
| 4 | 缺少链接（link）样式 | 三种来源都可能有链接 | 补充：Light 模式 #0066CC，Dark 模式 #4DA6FF，下划线 1px，点击态 opacity 0.7 |
| 5 | 缺少任务列表（task list）样式 | AI 生成常用 checkbox | 补充：checkbox 16x16dp，圆角 4dp，选中态跟随主题色，与首行文字居中对齐 |
| 6 | 缺少代码块语法高亮配色 | 技术笔记核心需求 | 补充 Light/Dark 两套语法高亮 Token（待确认是否有 HyperOS 官方配色，否则参考 GitHub 配色方案） |
| 7 | List 组件「行高」字段为空 | 研发无法实现 | 填入 text.primary 同款：16dp × 1.6 = 25.6dp（取整 26dp） |
| 8 | 声称支持响应式但无定义 | 不同屏幕宽度表现不可控 | 阅读区域 max-width 720dp，左右 padding lg(16)；compact 屏（<600dp）字号不变但间距缩减一级 |

### 建议补充

| # | 缺陷 | 影响 | 修复建议 |
|---|------|------|----------|
| 9 | `==text==` 高亮是非标准语法 | 导入内容可能不识别 | 明确标注为自定义扩展，定义 fallback |
| 10 | 缺少嵌套引用（多层 `>`）样式 | 导入内容可能有 | 补充嵌套层级递进规则 |
| 11 | 缺少脚注渲染规则 | 学术/长文场景 | 补充脚注标记和底部引用样式 |
| 12 | 缺少数学公式渲染规则 | 技术/学术笔记 | 补充 KaTeX/MathJax 渲染样式 |
| 13 | 无 Dark 模式切换过渡动画定义 | 切换体验生硬 | 补充 transition 时长和缓动曲线 |
| 14 | 缺少异常内容处理规则 | 脏 Markdown 渲染崩溃 | 定义 fallback：无法解析的内容以纯文本显示 |

## Approaches Considered

### Approach A: 最小可行 Demo（纯 HTML+CSS 静态页面）
- Effort: S
- 最快出活，但无法验证真实 Markdown 解析效果

### Approach B: 真实 Markdown 渲染 Demo（chosen）
- Effort: M
- 用 markdown-it + 自定义 CSS Custom Properties（即 `--color-text-primary` 形式的 CSS 变量，下文统称「CSS Token」）体系，做可粘贴/输入的实时预览页
- 能验证三种来源内容的一致性

### Approach C: 完整交付包（规范 + Demo + 验收 checklist）
- Effort: M-L
- 最完整但时间最长

## Recommended Approach

**Approach B: 真实 Markdown 渲染 Demo**

技术方案：
1. **补齐规范** — 修复上述 14 个缺陷，更新 Markdown 文档
2. **构建 Demo** — 单页 HTML + 资源文件夹（字体文件体积大，不适合内联），内嵌 markdown-it 解析器 + CSS Token 体系
   - markdown-it 核心 + 所需插件：markdown-it-task-lists（任务列表）、markdown-it-mark（高亮）、markdown-it-footnote（脚注，如纳入 MVP）、highlight.js（语法高亮）
   - 左侧：Markdown 编辑区（可粘贴任意内容）
   - 右侧：实时渲染预览（应用完整排版规范）
   - 顶部：Light/Dark 模式切换
   - 预置三组示例内容：用户手写风格、AI 生成风格、导入内容风格
3. **验收** — 三种来源内容在预览区渲染视觉一致，Light/Dark 模式均正确，所有 Markdown 元素样式符合规范定义

## Open Questions

1. 代码块语法高亮用什么配色方案？是否有 HyperOS 官方的代码配色 Token？
2. 表格在移动端窄屏下是横向滚动还是自适应缩放？
3. 图片是否需要支持点击放大？
4. 链接颜色是否跟随 HyperOS 主题色？
5. 数学公式和脚注是否在 MVP 范围内，还是后续迭代？

## Success Criteria

- [ ] 三种来源的 Markdown 内容在 Demo 中渲染视觉一致（同一元素的 font-size、line-height、margin 值在三种来源中完全相同）
- [ ] Light/Dark 模式切换后所有元素颜色正确、排版不崩（逐一对照 Color Token 表验证）
- [ ] 规范文档覆盖所有常见 Markdown 元素，研发无需反复确认（覆盖率 checklist：标题/正文/加粗/斜体/删除线/链接/图片/表格/代码块/引用/列表/任务列表/分割线/高亮，每项有明确的字号+字重+行高+颜色+间距定义）
- [ ] Demo 可以作为研发实现的视觉参考（研发对照 Demo 截图与 CSS Token 即可实现，无需额外沟通）

## Distribution Plan

## What I noticed about how you think

- 你说「不管用户自己写入的内容，还是 AI 生成的内容，还是导入的内容，都能有一个非常舒适的排版规范」— 你的出发点始终是用户体验的一致性，不是技术实现的便利性。这是设计师该有的立场。
- 你选了「先 demo 验证，再推全量」而不是直接甩文档给研发。这说明你知道规范文档本身不够，需要一个可感知的实物来推动落地。在大厂里，能跑的 demo 比 100 页文档有说服力。
- 你把 H0 定义为系统层页面标题而非 Markdown 元素，这个分层意识很好。说明你在想「什么该由系统控制，什么该由内容控制」，这是架构思维。
- 你对验收标准选了「全部」— 三种来源统一 + 暗黑适配 + 零歧义交付。标准高但合理，因为这三个不满足任何一个，规范就等于没用。
